### Decision: Git-native storage (events as JSON + per-event git repo)
**Date**: 2026-05-22
**Status**: Accepted
**Context**: mailproof needs a store for event state and an audit record. Options were a database as the source of truth vs. the git ledger as the source of truth.
**Decision**: Git-native. Event state = JSON + a per-event git repo whose hash-chained commits *are* the tamper-evidence. SQL/other consumers project a read-model; they do not replace the ledger.
**Consequences**: Limited ad-hoc query power and per-event repo overhead, but the proof is portable and offline-verifiable, and "the git ledger is the point" stays literally true. (Mirrors gitdone's original JSON+git decision.)

### Decision: Generic workflow only in v1 (sequencing scope)
**Date**: 2026-05-22
**Status**: **Superseded** (2026-05-24) by "Two generic modes: events + lean crypto sign-off"
**Context**: `completion.js` in gitdone mixes a generic ordered/parallel workflow engine with crypto declaration/attestation, strict signing, revoke, and threshold logic. The lift had to choose how much to bring.
**Decision**: Extract only the **workflow subset** — ordered/parallel/mixed steps. Crypto/attestation, strict signing, revoke, and thresholds stay in gitdone as policy on mailproof's hooks.
**Consequences**: Keeps the kernel lean and generic; consumers needing those modes implement them on the hooks. Risk: the workflow/policy seam must be clean enough that gitdone can rebuild its modes on top (proven in P2).

### Decision: Bundled Postfix/sendmail transport
**Date**: 2026-05-22
**Status**: Accepted
**Context**: Outbound email needs a transport. Options were a pluggable third-party provider (SendGrid/SES) vs. a self-hosted MTA lifted from gitdone's `outbound.js`.
**Decision**: Bundle Postfix/sendmail; opendkim signs outbound at the MTA. Not a pluggable third-party mail provider.
**Consequences**: More operator config (Postfix, opendkim, PTR, port 25 — to be documented in OPS.md) but full control, no vendor lock-in, and DKIM signing under the operator's own domain. Third-party providers are NO-GO (PRD §8.10).

### Decision: Email-only core verify/trigger path (with consumer side-effect carve-out)
**Date**: 2026-05-22
**Status**: Accepted
**Context**: Considered keeping a door open for non-email channels (webhooks/SMS) on the trigger or verify side.
**Decision**: Lock the core verify and trigger paths to email. DKIM/DMARC is what provides cold, third-party, offline verification with no pre-shared secret; webhooks (HMAC/mTLS) require a prior relationship and SMS has no real origin verification — neither preserves the "any two parties, no setup" property. Carve-out: consumers may fire their own side-effect notifications (webhook/Slack/SMS) from mailproof's `ingest()` result, carrying **no** mailproof verification guarantee.
**Consequences**: Email is the trust anchor by construction, not a swappable channel — this is the library's identity. Adopters needing a different channel need a different trust anchor (a different library). Integration reach is preserved via the side-effect carve-out. (PRD §8.1.)

### Decision: `receive.js` is not lifted — primitives + `createReceiver()` hooks
**Date**: 2026-05-22
**Status**: Accepted
**Context**: gitdone's `receive.js` orchestrates the four pillars, but it's tangled with gitdone's config singleton and crypto branches.
**Decision**: Do not lift the orchestrator. mailproof exposes the four pillars as primitives plus an optional `createReceiver()` with policy hooks; each consumer writes its own thin glue.
**Consequences**: Consumers write a little wiring, but the kernel stays decoupled from any one app's policy and config. "Mechanism, not policy" holds at the orchestration layer too.

### Decision: Accept-with-flag (commit every reply, gate the transition)
**Date**: 2026-05-22
**Status**: Accepted
**Context**: How to treat replies that fail routing/trust (wrong participant, unverified DKIM, out-of-order).
**Decision**: Commit *every* inbound reply to the ledger as an audit record; a separate `counted` flag records whether it advanced state. Routing and trust gate the state transition, never the commit.
**Consequences**: Complete, tamper-evident audit trail including rejected attempts (POC: 8 commits, 2 counted). The ledger answers "what was received" independently of "what counted."

### Decision: Config injection by bound per-pillar factories (no config singleton)
**Date**: 2026-05-23
**Status**: Accepted
**Context**: gitdone modules read a `config` singleton (`config.dataDir`, …) evaluated from `process.env.GITDONE_*` at `require()` time, with gitdone-specific defaults baked in (`git-done.com`, `/var/lib/gitdone`). P1 had already decided to "replace the config singleton with injected config" (DESIGN P1) for three reasons: a *library* can't assume the consumer's env-var names, paths, or domain; a load-time singleton is hidden coupling (gitdone's own tests must `delete require.cache` to retest it); and one-singleton-per-process blocks two stores with different `dataDir`s. `outbound.js` (module 4) already injects config **per call** (`binary`, `domain`, `footer`). The open question for the storage pillar was the *form* of injection: per-call vs. a bound factory.
**Decision**: Inject **fixed-per-deployment** config (`dataDir`) via a **bound factory per pillar** — `createEventStore({ dataDir })`, `createGitrepo({ dataDir })`, etc. — each naming the value once and returning the bound primitives. `create({ dataDir, domain, sendmailBin, otsBin? })` composes these factories and remains the single runtime source of truth. Config that legitimately **varies per call** (outbound's `domain`/`footer`, which differ per message) stays per-call. The discriminator is the nature of the value, not uniformity of style.
**Consequences**: Banning the singleton did not remove the single source of truth — it moved it from *implicit* (ambient env, read at import, settable from anywhere) to *explicit* (the `create`/factory call). The factory **preserves** that single explicit binding point; per-call injection of a fixed value would re-fragment `dataDir` across every call site (split-brain risk) — so the factory is the consistent finish to the ban, not a new preference. Trade-off: a mild surface asymmetry (storage = factory, outbound = per-call), justified by the fixed-vs-varying distinction. Simplest at the call site, one source of truth — the project's standing tie-breaker.

### Decision: Two generic modes — events + lean crypto sign-off (supersedes "generic workflow only")
**Date**: 2026-05-24
**Status**: Accepted
**Context**: The original lift scoped mailproof to *generic workflow only* and put all of gitdone's crypto declaration/attestation in the NO-GO table (§8.2). A design conversation reframed mailproof's identity: gitdone's two proven shapes of DKIM-verified multi-party coordination — **events** (named participants, ordered steps) and **attestation/sign-off** (verified signers toward a goal) — are both *generic mechanisms*, not domain policy. The goal became "publish gitdone's engine as a public library" ("same like gitdone but public"), with gitdone-the-product keeping only its web/branding/hosting wrapper. The risk was re-importing completion.js's heavy tail (strict signing, revoke, three dedup variants, attestor_progress, PII redaction — ~500 of its 733 lines) and bloating the kernel.
**Decision**: mailproof offers **two generic coordination modes** on the shared DKIM-verified-email + notary substrate (PRD §1, §4.2): **events** (`type:"workflow"`, the m6 engine) and **crypto sign-off** (`type:"crypto"`). Crypto is **one parameterized engine, not three modes**: `signers` (explicit email list — manually added — *or* open via a shared `attest+{id}@`-style address), `threshold` (1 = single-signer declaration; N = count-to-goal), and an optional single `requiredDocHash` (the "email + doc" two-layer, via the notary). A sign-off counts iff DKIM-verified, sender ∈ signers (or any, if open), a *distinct* sender not already counted, and (if set) the doc hash matches; complete at distinct-count ≥ threshold. **The "link" is a shared email address, NOT a web link/app/webhook** — discovery may be a shared address, but the signing act and its verification are always DKIM email.
**Consequences**: **Lean by exclusion** — explicitly OUT (gitdone policy, §8.2–8.4): `revoke`, `latest`/`accumulating` dedup (distinct-only in core), multi-doc strict manifests + `attach` channel, attestor-PII redaction, magic-link/web flow, branding. Estimated ~100–120 lines vs. completion.js's ~500. **No rework**: the m6 events engine is unchanged; crypto is an *additive* second engine (next module, "m6.7"), and m7's `create()/ingest()` routes by `type`. Identity widens from "sharp primitive" to "headless gitdone" — recorded here as a *deliberate* reversal, not drift. Trade-off: a broader API surface and a fuzzier one-line pitch, accepted for the much higher reuse value of shipping both proven mechanisms. (PRD §1, §4.2, §8.2–8.4, §8.12; supersedes "Generic workflow only in v1".)

### Decision: Crypto sign-off engine (m6.7) — three boundary forks resolved
**Date**: 2026-05-24
**Status**: Accepted
**Context**: Implementing the lean crypto engine (above) surfaced three forks the two-mode decision didn't pin, raised before coding (per the standing "agree the boundary first" rule): (1) is the verified-trust gate hardcoded or a per-event knob; (2) does gitdone's initiator self-reply exclusion survive into the unified engine; (3) does m6.7 also add the `attest+{id}@` router tag, or just the engine.
**Decision**: (1) **Trust hardcoded to `verified`, no per-event `minTrust`** — crypto is the high-assurance mode and letting a forwarded/authorized reply stand as a signature is a footgun; graded trust stays a workflow-only feature (per-step `minTrust`). (2) **Keep the initiator-exclusion in crypto** (`initiator_self_reply`): the initiator may reply and is committed for audit, but their reply never counts as a verification (anti-self-dealing — you can't be both orchestrator and verifier). This diverges from my "drop it" recommendation on the user's explicit call. **Events differ** — there the initiator *may* be a counted participant (m6 unchanged). (3) **Pure engine only**, mirroring how m6 landed: `src/crypto.js` + unit tests + SPEC §3.1/§4 crypto shape; the net-new `attest+{id}@` router tag and SPEC §2 promotion **defer to m7b assembly**, where the route is exercised end-to-end.
**Consequences**: The engine stays pure (no I/O, no crypto): sender-identity resolution is the orchestrator's, passed as precomputed `signer_match` / `is_initiator` booleans — the same boundary as workflow's `participant_match`, so plaintext never reaches the engine or ledger (SPEC §6). ~150 lines, distinct-count-to-threshold only; gitdone's `dedup`/`revoked_senders`/`reference_docs`/`mode` are inert if present (a regression test proves it). Trade-off of (1): a consumer wanting weaker-than-verified sign-off must layer it on the hooks, not the kernel — accepted, since that would dilute the mode's guarantee. (PRD §4.2; SPEC §3.1; CHANGELOG m6.7.)

### Decision: Document notary in scope (auto-hash + `verifyDocument`); doc-as-login is NO-GO
**Date**: 2026-05-24
**Status**: Accepted
**Context**: A design conversation explored using attached documents as a verification layer. Ideas ranged across a per-step hash *gate*, a "claim-check" retrieval credential ("send me a doc we exchanged + your email to unlock X"), and even a login / second factor (magic-link + doc). The question was how much, if any, belongs in the kernel, and whether it collides with the §8.3 strict-signing NO-GO. The user explicitly asked to push back on bloat and admitted no concrete threat model — "I just thought it's neat."
**Decision**: The kernel gains a **minimal document notary**, distinct from gitdone's strict-signing: (1) **mandatory** SHA-256 hashing of *every* inbound attachment into its commit (hashing is always on; whether to *use* a doc as a verification layer is the consumer's choice), and (2) a read-only `verifyDocument(doc)` lookup returning the matching commit(s) with trust/sender/counted info. The document is framed as a **proof-of-participation receipt, not a secret** — the DKIM-verified sender is the trust factor; the doc adds tamper-evident binding/context only. **Deferred** (not NO-GO, built when a consumer needs them): outbound-attachment hashing, `listDocuments`, retrieval-gating. **NO-GO**: document-as-login / second factor (PRD §8.8) — a doc over email is a weak, clunky factor (same inbox; worse than TOTP/passkeys) and the verified sender is already the factor; mailproof is not an auth provider.
**Consequences**: Distinct from §8.3 — strict-signing (pre-registered `reference_docs` manifests, per-attestor progress, completion-gating) stays gitdone policy; the notary only *records and verifies*, it never gates workflow completion, so the completion engine (m6) is untouched by this. Delivers mailproof's "docs hashing, verifiable later" mission with two small additions (m6.5: auto-hash + `verifyDocument`) on the existing ledger, which already stores `attachments[]`. Enumeration of acceptable docs is **not** a leak (SHA-256 is preimage-resistant — a hash list can't produce a file); the real weak point is possession-secrecy, which is why the receipt framing (not password) is the honest one. Risk: scope pressure to grow the deferred items or revive login — guarded by §8.8 and this entry. (PRD §4.1.)

### Decision: git ledger uses the `git` binary via child_process, not `simple-git`
**Date**: 2026-05-23
**Status**: Accepted
**Context**: gitdone's `gitrepo.js` uses `simple-git@^3.20.0`. Lifting it into mailproof would add the **first external runtime dependency**, so AGENT_RULES requires running the External Dependency Checklist first. Audit of the module showed `simple-git` is used for exactly five git operations (`init`, `config`, `add`, `commit`, `status`); `listCommits` reads the filesystem, never `git log`. The two behaviors actually consumed are the commit SHA (`rev-parse HEAD`) and staged-file detection for the byte-strict no-op check in `syncEventJson` (`diff --cached --name-only`).
**Decision**: Do **not** add `simple-git`. Lift `gitrepo` onto the stdlib `git` binary via `child_process.execFile`, through a ~6-line `git(cwd, args)` helper. Note that `simple-git` is itself a thin wrapper that spawns the same `git` binary — it adds output-parsing convenience, not capability or portability; both approaches require `git` installed.
**Consequences**: **Necessity gate fails** for `simple-git` — the five ops are well under the AGENT_RULES <100-line stdlib bar. Zero runtime deps so far, preserving the ≤3 budget for `mailauth` + `mailparser`, which are security-critical (DKIM / untrusted-MIME parsing) and genuinely *require* vetted libraries. Consistent with `outbound`'s zero-dep `sendmail` shell-out and DESIGN's "stdlib + git binary only" P0 baseline. Cost: ~2 commands of hand-parsed git output (`rev-parse`, `diff --cached`), verified against real git by the lifted `gitrepo` characterization tests. The four simple-git conveniences we forgo (empty-commit guard, error surfacing, identity ordering, `index.lock` retries) are handled explicitly — the lock case by m5a's per-event `withEventMutex`.

### Decision: First external deps — `mailauth` + `mailparser` accepted (m7a)
**Date**: 2026-05-26
**Status**: Accepted
**Context**: m7a lifts the inbound decoder (DKIM/DMARC/ARC authentication + MIME parsing), which mailproof had so far avoided — every prior module ran on stdlib + the `git` binary (zero runtime deps). This is the point the ≤3-dep budget gets spent. The AGENT_RULES External Dependency Checklist had to clear before installing. A second finding: the stash labelled m7a a `verify.js` lift, but reading the source showed `verify.js` is gitdone's `verify+{id}@` *endpoint* (forwarded-email report + reverify-against-archived-PEM); the actual inbound authenticate+parse core lives in `bin/receive.js` (the orchestrator, not lifted).
**Decision**: Add **`mailauth`** (DKIM/DMARC/ARC) and **`mailparser`** (MIME) as the first 2 runtime deps. Checklist verdict: **Necessity** — DKIM/MIME of hostile input is thousands of lines and security-critical, so a vetted library is *required* (rolling our own is the forbidden "clever" move); **Maintained/Established** — both by Andris Reinman (Nodemailer author), actively published, the de-facto Node implementations; **Lightweight** — moderate transitive trees (undici, iconv-lite, tldts…), accepted as the *irreducible* cost of correct mail handling, not incidental bloat, and the transitive deps are themselves first-tier; **no new supply-chain surface** — they are the same two packages gitdone already ships. Extract the core as `src/parse.js` (`authenticateMessage` + `parseMessage`), **not** a `verify.js` lift; gitdone's `verify+@` endpoint, DSN bounce handling, and durable dkim-archive reverify are **deferred** to later modules.
**Consequences**: Runtime deps 0 → 2, within the ≤3 budget (the git ledger still uses the `git` binary, not `simple-git`, per 2026-05-23). `authenticateMessage` pins `trustReceived: false` as the one documented, safe call into mailauth (forged `Received`/`Authentication-Results` headers are the obvious attack); its `resolver` injection keeps tests offline and leaves room for the future verify+@ endpoint to re-check against an archived key. `parseMessage` completes the notary **capture** half (deferred from m6.5) by auto-hashing every attachment through `hashDocument` — one fingerprint format kernel-wide. Trade-off: the install footprint grows, accepted because correctness in a security-critical domain outranks a smaller `node_modules`. (CHANGELOG m7a; supersedes the stash's "verify.js authenticate core" label.)

### Decision: Privacy/erasure stance — durability-first; per-record RTBF is NO-GO
**Date**: 2026-05-27
**Status**: Accepted
**Context**: Before m7b freezes the commit format, the question arose whether to bake in a right-to-be-forgotten path — specifically, letting an event participant self-revoke/erase their own ledger records by default. mailproof already minimizes (SPEC §6: no plaintext sender/subject/body/Message-ID; only a salted `sender_hash` + plaintext `sender_domain`). The salt is the only link from hash→person; today it is **public and committed in-chain** (`event-store` generates it; `gitrepo.syncEventJson` mirrors the whole `event.json` — salt included — into the immutable per-event repo). So records are pseudonymous, not anonymous, and not erasable without breaking the chain.
**Decision**: **Durability-first.** mailproof's value is a tamper-evident, non-repudiable, offline-verifiable multi-party proof — all three of which per-record erasure attacks (deleting a commit rewrites every descendant SHA; erasing a counted reply turns "3 of 3 signed" into "2 of 3"; the salt must travel in the repo for "git clone = the proof"). Therefore: (1) privacy = data minimization (already a core invariant) + the "receipt, not secret" framing (the DKIM-verified sender is the trust factor; the hash only binds); (2) per-record self-revoke / erasure is a **NO-GO** (PRD §8.16) — it contradicts non-repudiation and tamper-evidence, and GDPR's erasure right carves out the establishment/exercise/defence of legal claims and legal obligations, under which a contractual sign-off record plausibly falls (not legal advice); (3) the ONLY erasure lever is coarse and opt-in: destroying an **entire event** (the consumer deletes `event.json` + the repo) also destroys that event's salt, making any residual exported hashes unlinkable — whole-event crypto-shredding falls out of event deletion, **nothing new to build**. The public in-chain per-event salt stays, chosen over a destroyable out-of-chain salt because verifiability ↔ forgettability are the same dial and public offline verification (the `verify+@` proof-of-participation) is the product.
**Consequences**: No new erasure subsystem; m7b's commit format is unaffected. The privacy story is honest: minimized + pseudonymous + durable by design — not anonymous or per-record forgettable. **Rejected alternatives**, recorded so they aren't relitigated: (a) *forgettability-capable* — a destroyable, out-of-chain, per-participant salt enabling crypto-shred RTBF, buildable later if a consumer's legal posture demands it, at the cost of public offline verification; (b) *tombstone/logical-delete* — append-only redaction commits that hide records from projections but leave the hash + public salt intact, so not true anonymization. (PRD §8.16; SPEC §6.)

### Decision: Email triggers — occasions are kernel (+ neutral templates), bodies are policy; roadmap m7c verify → m7d triggers
**Date**: 2026-05-27
**Status**: Accepted
**Context**: After the m7b assembly + m7c-1 landed, the user flagged that gitdone's email triggers are rich (`notifications.js`: `notifyWorkflowParticipants` incl. reminder mode, `notifyEventCompletion` with embedded receipts, `notifyOrganiserOfActivation`, `notifyInitiatorOfSigningProgress`, `notifyOrganiserOfStepProgress`, `notifyProofAnchored`, `notifyDeclarationSigner`; plus `sweep.js` for reminders/deadlines/archive and `dsn.js` for bounces) and that mailproof's Commit-C trigger layer emits only three (`advance`/`ack`/`completion`). The tension: §8.6 makes branded email *bodies* a NO-GO, but the north-star acceptance test ([[mailproof-rebuilds-gitdone-acceptance-test]]) requires a consumer to be able to rebuild gitdone — so the full set of trigger *occasions* must be expressible without re-deriving kernel state.
**Decision**: Split the trigger pillar on **occasion vs body**. *Occasions* (the events that warrant an email — derivable from a state transition, a timer, or an inbound bounce) are **mechanism**: mailproof emits every occasion it can derive, each as a `kind` with a **simple neutral default template**, all overridable through the single `composeNotification(ctx)` hook keyed by `kind`. *Bodies/branding/prose* stay **policy** (§8.6 intact). This requires absorbing three deferred subsystems into the kernel: **`sweep()`** (time-driven `reminder`/`overdue`/`archived`), **`dsn.js`** (inbound bounce → `bounce`), and small **activation** + **edit-renotify** hooks; plus richer `completion` ctx (the counted commits/receipts, so a consumer can embed proof without mailproof owning the body) and `proof_anchored` (rides on OTS verify). The Commit-C `composeNotification`/`deliver()` seam already supports this — it is extended with `kind`s, not reworked. **Stays policy:** gitdone's strict-attach "docs needed" nudge and signing-progress UX prose.
**Consequences**: Roadmap ordering set (user's call): finish **m7c** (the verification surface — `verify+`/`reverify`/OTS-verify/e2e) FIRST, because the gitdone reconverge (P2) hard-depends on offline verifiability, THEN a dedicated **m7d** triggers milestone that absorbs all kernel-derivable occasions + neutral templates. Reaffirms §8.6 (no branding in the kernel) while widening the trigger pillar from "fire the next email" to "emit every occasion, neutrally templated." `sweep()` and `dsn` graduate from "deferred" to "scheduled (m7d)". (CHANGELOG m7c-1; PRD status/roadmap; supersedes the stash's "DEFER sweep()/DSN" line — they are now roadmap, not dropped.)

### Decision: OTS-proof verification = `ots upgrade` + `ots info`, never `ots verify`
**Date**: 2026-05-27
**Status**: Accepted
**Context**: m7c-4 (the last verify slice) needed a way to confirm a committed `.ots` proof is genuinely Bitcoin-anchored, not just calendar-pending. The obvious primitive is `ots verify`. But reading gitdone's working wrapper — which I initially missed in `app/src/ots.js` (stamp-only) and the user pointed to in `app/bin/ots-upgrade.js` — showed gitdone deliberately avoids `ots verify`: it queries the calendar servers *live*, so it reports "verified" for a proof whose attestation hasn't actually been folded into the file. If the calendars died, that proof would be unverifiable offline despite having "passed".
**Decision**: Port gitdone's design as the kernel primitive: `createOts().upgradeProof(abs)` runs `ots upgrade` in place and treats the proof file's **sha256 changing** as the authoritative "now anchored in Bitcoin" signal (the calendar attestation was merged into the file — now offline-verifiable forever); `readBlockHeight(abs)` reads the anchored block via `ots info` (parses the binary proof **locally, no network**); and the pure `parseOtsBlockHeight` (re-exported) is the testable output parser. `ots upgrade`'s non-zero exit on a still-pending proof is modelled as `{ pending: true }`, **not** an error (accept-with-flag). We never call `ots verify`.
**Consequences**: The trust anchor is durable and offline (the upgraded `.ots` carries the Bitcoin proof), matching mailproof's "git clone = the proof" thesis — not a live calendar query. Kernel/policy split (consistent with the triggers-occasions entry above): the **primitive** is kernel (`createOts`); the **scheduler** that walks an event's proofs, records `ots_anchored` into the ledger, and emits the `proof_anchored` occasion is policy/glue, deferred to **m7d** (it's gitdone's `processRepo`/`run` cron, not the wrapper). Testability: the pure parser is unit-tested across the opentimestamps-client output shapes; the subprocess wrappers are integration-tested for the binary-missing path + a real-`ots` non-proof smoke test (deterministic, no network); the fully-confirmed-proof path stays deployment-verified (needs ~1h of Bitcoin confirmations). With m7c-4 the 9-point [[mailproof-rebuilds-gitdone-acceptance-test]] checklist is fully covered. (CHANGELOG m7c-4; SPEC §4 tamper-evidence.)

### Decision: pending-activation TTL/reminder is consumer policy, not a kernel occasion (m7d-2)
**Date**: 2026-05-27
**Status**: Accepted
**Context**: m7d-1 deferred two of gitdone `sweep.js`'s passes to m7d-2: the never-activated TTL-DELETE (purge events created but never activated, after ~72h) and the pending-activation about-to-expire REMINDER. Implementing m7d-2 (the activation lifecycle) forced the boundary question the deferral left open: are these kernel occasions?
**Decision**: **No — both are consumer policy.** (1) The TTL-delete is destructive and the privacy decision (durability-first; 2026-05-27) already names whole-event deletion as the consumer's only erasure lever — the kernel must not auto-delete an audit record. (2) The about-to-expire reminder presupposes a SELF-SERVICE activation UX (the organiser must take an action — gitdone's magic-link click — or the event expires), which mailproof deliberately dropped when it trimmed gitdone-web's magic-link activation (m5a). In mailproof, activation is a programmatic `activateEvent()` call; a draft sitting un-activated is the consumer's intent, not a countdown. So m7d-2 ships only the two occasions that ARE kernel-derivable from a real transition: `activation` (first `activateEvent` → kickoff to initially-eligible participants/signers) and `reassigned` (`editEvent` moves a participant onto a currently-eligible step). A consumer that wants gitdone's draft-cleanup composes it on `listEventIds()` + `loadEvent()` + whole-event deletion, and a draft-idle nudge on `listEventIds()` + the shared notifier — no kernel state to add.
**Consequences**: Supersedes m7d-1's "deferred to m7d-2" note for those two passes (they are reclassified, not built). Keeps the kernel from re-absorbing the magic-link self-activation model it shed, and the no-kernel-deletion invariant intact. The occasion taxonomy is now advance/ack/completion (ingest) + overdue/archived (sweep) + activation/reassigned (organiser actions); bounce + proof_anchored remain (m7d-3/4). Consistent with the occasions-are-kernel/bodies-are-policy boundary (2026-05-27): an occasion must derive from a transition mailproof actually models. (CHANGELOG m7d-2; PRD status.)

### Decision: initiator-command boundary — `remind+` reuses existing occasion kinds; `stats+` returns a snapshot, no outbound
**Date**: 2026-05-28
**Status**: Accepted
**Context**: m7d-5b lifted the two initiator commands the router already parses (`remind+{id}@`, `stats+{id}@`) into `ingest()`. Two real boundary calls fell out. (1) **What `kind` does `remind+` fire under?** A new `kind:'remind'` adds a taxonomy entry every body composer must template separately; reusing the existing kinds keeps the body-hook surface narrower. The user picked the reuse shape ("reuse advance/ack + reminder flag" — fewer kinds is simpler, matches gitdone's `notifyWorkflowParticipants({reminder:true})`). The literal "advance/ack" pair didn't survive the crypto side: `ack` is a post-reply receipt and inverts the semantics ("ack of something that hasn't happened") for the "still need your signature" case. (2) **Does `stats+` send a reply?** gitdone's `stats+` calls `statsBody(event)` and sends a rendered text reply via sendmail; mailproof's body boundary (§8.6) says rendered prose is policy.
**Decision**: (1) `remind+` reuses the existing kinds with `ctx.reminder = true`: workflow remind fires `kind:'advance'` for every currently-eligible step (same recipients as a cascade-advance — the principle is "same body path the consumer already templated"); **crypto remind fires `kind:'activation'`** for every listed signer that hasn't yet signed (same recipients as the activation kickoff — "still need your signature" IS the kickoff message reframed). One body hook, two existing kinds, `reminder` distinguishes inside the composer. Open crypto has no roster → remind is a no-op (the initiator owns distribution); already-complete or archived events short-circuit with `reason:'already_complete'`, no sends. (2) `stats+` returns a structured kernel snapshot on the `ingest()` result AND sends a **neutral default reply** to the initiator as a new `kind:'stats'` (default body = a plain ASCII dump of the snapshot — gitdone parity out of the box). The snapshot rides on `ctx` so a consumer's `composeNotification` keyed on `kind:'stats'` overrides the prose — same body-hook seam as every other kind, branding stays policy (§8.6). The snapshot shape is `loadEvent(id)` reshaped into a stable surface: `{eventId, type, title, status, activated_at, archived_at, completed_at, flow?, steps?, threshold?, open?, signers?, signatureCount?}`. (Originally `stats+` was scoped snapshot-only with no outbound, on the rationale that rendered prose is policy; the user picked the parity-out-of-the-box shape so the consumer doesn't have to wire compose+sendmail just to get a working `stats+`.)
**Consequences**: Neither command commits to the ledger (operational, like `bounce`). Auth matches gitdone's `authenticateInitiatorCommand` — DKIM-verified + envelope sender == event.initiator (PRD §6.4).
